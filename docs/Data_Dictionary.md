# Data Dictionary

---

# Overview

This document defines every dataset, table, column, and business entity used within the Insurance Business Intelligence Dashboard.

The objective of this document is to ensure consistency in reporting, simplify maintenance, and provide a single source of truth for business users and developers.

---

# Data Model Overview

The solution follows a Star Schema.

```
                  Dim_Date
                      |
                      |
Dim_Agent ---- Fact_Business_Raw ---- Dim_RM
                      |
                      |
             Fact_Commitment_Raw
```

---

# Fact_Business_Raw

## Purpose

Stores actual business generated by agents.

Each row represents one business transaction.

---

### Primary Key

No single primary key.

Business uniqueness is created using:

- Booking Date
- Partner Code
- Company
- Policy Number (if available)

---

### Important Columns

| Column | Data Type | Description |
|---------|-----------|-------------|
| Booking_Date | Date | Date business was booked |
| Business_Month_Key | Date | Month key used for reporting |
| Partner_Code | Text | Unique Agent Code |
| Partner_Name | Text | Agent Name |
| RM_Code | Text | Relationship Manager Code |
| RM_Name | Text | Relationship Manager |
| Department_Name | Text | Department |
| LOB | Text | Line of Business |
| Insurer_Name | Text | Insurance Company |
| Product_Name | Text | Product |
| NetPrem | Decimal | Net Premium |

---

# Fact_Commitment_Raw

## Purpose

Stores business commitments assigned to agents.

Each row represents one commitment.

---

### Important Columns

| Column | Data Type | Description |
|---------|-----------|-------------|
| Effective_Date | Date | Commitment effective date |
| Business_Month_Key | Date | Reporting month |
| Agent_Code | Text | Agent Code |
| Agent_Name | Text | Agent Name |
| RM_Code | Text | RM Code |
| RM_Name | Text | RM Name |
| Commitment_Amount | Decimal | Business commitment |

---

# Data Quality Rules

- RM Code must never be blank.
- Partner Code must be unique within dimensions.
- Dates must be valid.
- No duplicate dimension keys.
- Net Premium cannot be negative.
- Commitment cannot be negative.
